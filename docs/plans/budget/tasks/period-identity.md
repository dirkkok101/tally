## Recipe

### Acceptance Checks

- A valid year/month/ZAR resolves startInclusive on month day one and endExclusive on next month day one; every period-bearing operation requires this explicit input.
- One TimeProvider-derived DateOnly classifies start-inclusive/end-exclusive periods as Current, Future, or Closed and cannot be overridden by JSON or stored columns.
- Equivalent explicit period inputs produce equal BudgetPeriod values and identical boundaries regardless of invocation time or caller locale.
- Invalid month, non-ZAR, non-calendar, overflow, or omitted period returns stable validation before state access.

### Failure Criteria

- Do NOT infer the requested period from invocation time, persist period state, accept caller lifecycle state, or use UTC timestamp comparisons for calendar eligibility.
- Do NOT add rollover, reopen, arbitrary ranges, multi-currency, or policy-engine behavior.
