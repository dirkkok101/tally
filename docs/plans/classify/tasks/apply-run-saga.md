## Recipe

### Acceptance Checks

- Before the first Ledger call, apply.run locks the run, repeats apply_preflight for the entire selected set, and rejects all items on any missing, stale, ineligible, allocation, revision, category, authority, or reason mismatch.
- For each deterministic transaction-ID item, exact operation, target, expected allocation, revisions, reason, request fingerprint, and derived idempotency key are durable before the Ledger call.
- Ledger atomically asserts assignment absence or exact correction allocation and revisions; CLASSIFY then records applied, already_applied, rejected, failed, or unresolved before advancing.
- Identical replay returns prior terminal results; conflicting identity reuse preserves original results and returns stable conflict.
- Every injected crash window resumes only planned or unresolved items, never calls Ledger again for terminal rows, and creates zero duplicate allocations.

### Failure Criteria

- Do NOT hold a CLASSIFY SQLite transaction across Ledger calls or claim a distributed transaction - rejected by DD-CLASSIFY-APPLY-SAGA.
- Do NOT call Ledger before durable exact intent or retain only an aggregate receipt.
- Do NOT replay terminal items or regenerate a request, key, reason, allocation expectation, or revision on resume.
- Do NOT continue with a partially stale selection or weaken Ledger errors.

### Expected Outputs

- RunClassificationApplyCommand
- ClassificationApplyRunStore
- ClassificationApplyLock
- ApplyReplayPolicy

### Constraints

- One owner-only per-run OS lock; Ledger call occurs outside CLASSIFY transaction; all async boundaries propagate CancellationToken.

### Notes

- The crash matrix must cover commit-before-result and result-before-CLASSIFY-durability windows explicitly.
