## Recipe

### Acceptance Checks

- Every input ordinal yields exactly one suggestion, no_suggestion, conflict, or stale result and aggregate totals equal input count.
- One match or multiple same-category matches produce one suggestion naming every contributing rule version; incompatible categories produce conflict with no selected category.
- Identical fingerprints produce semantically identical ordered outcomes and evidence under repeated and permuted property cases.
- EvaluationFingerprint covers contract, projection, store generation, snapshot and expiry, category lifecycle, normalization, rule-set, and ordered item revisions.
- MatchEvidence retains only rule, condition, field, predicate, and normalized-value hash references.

### Failure Criteria

- Do NOT select a winner by rule priority or input order - rejected by DD-CLASSIFY-DETERMINISTIC-EVALUATION.
- Do NOT publish partial streamed outcomes or perform storage or Ledger mutation from the engine.
- Do NOT retain full projection payloads or omit all historical evidence - both alternatives were rejected by DD-CLASSIFY-DETERMINISTIC-EVALUATION.
- Do NOT introduce random, clock, culture, network, or host-session inputs into matching.

### Expected Outputs

- ClassificationEngine
- EvaluationFingerprint
- CanonicalClassificationHasher
- Bounded MatchEvidence

### Constraints

- Pure synchronous domain evaluation; all ordering and limits are explicit inputs.
