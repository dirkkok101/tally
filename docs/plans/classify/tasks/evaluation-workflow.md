## Recipe

### Acceptance Checks

- The command obtains one complete compatible ClassificationEvaluationInput before creating an evaluation ID.
- One SQLite transaction persists the evaluation, fingerprints, ordered outcomes, bounded MatchEvidence, and totals equal to input count.
- Each outcome is the exact ordered ClassificationEngine result for the immutable active rule set and input fingerprint.
- Identical idempotency and request return the stored result; conflicting reuse returns a stable conflict.
- Ledger facts, relationships, and allocations are byte-equivalent before and after evaluation.

### Failure Criteria

- Do NOT publish or persist a partial evaluation - rejected by DD-CLASSIFY-DETERMINISTIC-EVALUATION.
- Do NOT bypass ClassificationEvaluationInputLoader or re-evaluate Ledger eligibility.
- Do NOT persist raw input descriptions or unrelated financial fields.
- Do NOT invoke assignment or correction from evaluate.

### Expected Outputs

- EvaluateClassificationCommand
- ClassificationEvaluationStore
- Atomic evaluation result mapping

### Constraints

- All page and store async boundaries propagate CancellationToken; the public result is bounded and source-generated.
