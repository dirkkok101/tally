## Recipe

### Acceptance Checks

- ClassifyExtensions explicitly composes contract, state, corpus, rule, evaluation, apply, feedback, and recovery registrations and Program adds it without reflection scanning.
- Published schema inventory contains exactly twelve C12 descriptors twice with byte-stable semantics and no hidden alias, HTTP route, background operation, or private data read.
- Every valid operation emits exactly one versioned business envelope; every malformed, unknown-field, not-found, conflict, stale, compatibility, resource, and Ledger failure emits one stable module error and documented exit.
- Mutation operations reject missing actor or idempotency before state change; incompatible CLASSIFY or Ledger versions fail before financial-data read or mutation.
- A linux-x64 Native-AOT publish completes with source-generated JSON and zero trim, reflection, dynamic-code, build warning, or descriptor-registration gaps.

### Failure Criteria

- Do NOT register handlers through reflection scanning or hidden conventions - per DD-CLASSIFY-APPLICATION-ARCHITECTURE and ADR-CORE-0021.
- Do NOT add HTTP, a service, generic invoke, aliases, prompts, serialized diagnostics, or unversioned contracts.
- Do NOT mark this gate green when any descriptor lacks a concrete registered handler or focused process test.
- Do NOT update external dependency validation status here; the final module gate owns evidence-based status changes.
