## Recipe

### Acceptance Checks

- Given purpose=evaluation, the public projection returns only active transactions requiring an independent category decision and preserves complete snapshot order across pages.
- Given purpose=apply_preflight and bounded transaction IDs, every selected identity is returned or reported missing with assignable, correctable, or ineligible state plus current allocation and lifecycle revisions.
- Given an assignment with an existing allocation or a correction whose expected allocation or revision drifted, Ledger returns a stable stale-precondition result and appends no allocation.
- Given an incompatible projection or mutation contract version, descriptor validation fails before financial-data read or mutation.

### Failure Criteria

- Do NOT add a private classification query or direct CLASSIFY access to Ledger storage - rejected by DD-CLASSIFY-LEDGER-PUBLIC-PROJECTION.
- Do NOT re-infer transfer, refund, lifecycle, categorization, or allocation eligibility outside Ledger - rejected by DD-CLASSIFY-LEDGER-PUBLIC-PROJECTION.
- Do NOT permit correction without the exact expected active allocation and projection revisions - the race would violate DD-CLASSIFY-LEDGER-PUBLIC-PROJECTION.
- Do NOT mutate transaction source facts or historical allocation rows.

### Expected Outputs

- Versioned classification_v1 projection contracts
- Ledger-owned mutation-state and revision preconditions
- Focused projection and stale-write tests

### Constraints

- Modify the Ledger Actuals and allocation slices produced by PLAN-LEDGER-V1; do not create parallel implementations.
- Preserve source-generated JSON and Native-AOT compatibility.
