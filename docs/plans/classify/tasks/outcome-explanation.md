## Recipe

### Acceptance Checks

- Suggestion, no-suggestion, and conflict explanations return exact retained evaluation, transaction, category, normalization, rule-set, contributing rules, allowed matched fields, and stable reason without unrelated payloads.
- Unknown outcome and known outcome with unavailable retained evidence return distinct stable statuses and never reconstruct evidence from current state.
- Transaction, allocation, transfer, refund, category lifecycle, contract, projection, store generation, snapshot, normalization, and rule-set changes each name the changed fingerprint dimension and allow only re-evaluation.
- A same-identifier active category rename preserves validity and returns the current display name; archive or reactivation stales the outcome.
- Stale, conflict, and no-suggestion outcomes remain unappliable.

### Failure Criteria

- Do NOT reconstruct missing MatchEvidence from current Ledger or rule state - rejected by DD-CLASSIFY-DETERMINISTIC-EVALUATION.
- Do NOT treat display-name rename as category identity change.
- Do NOT expose raw descriptions, normalized values, corpus rows, hidden rule payloads, or full requests.
- Do NOT offer correction or forced apply as the safe action for stale evidence.

### Expected Outputs

- GetClassificationOutcomeQuery
- ClassificationStalenessPolicy
- Bounded outcome result mapping

### Constraints

- Query reads retained CLASSIFY evidence and public Ledger display/lifecycle state only; it performs no durable mutation.

### Notes

- Staleness comparison is field-complete against EvaluationFingerprint; tests enumerate every PRD trigger.
