## Recipe

### Acceptance Checks

- On linux-x64 the CLASSIFY directory is 0700 and database, WAL, SHM, lock, report, and temporary files are 0600; unsupported ownership or permissions block persistence.
- Startup applies ordered user_version migrations with foreign_keys ON, WAL, synchronous FULL, bounded busy handling, and a single supported owner writer.
- Rule, validation, evaluation, apply, feedback, abandonment, cleanup, and operation-idempotency tables match the nine CLASSIFY data-model contracts.
- Immutable versions, outcomes, terminal item results, feedback, and lifecycle rows reject UPDATE and DELETE; guarded transitions require the expected prior state.
- Reusing an idempotency key with the same request returns the stored result; a different request returns a stable conflict.

### Failure Criteria

- Do NOT share the Ledger database, use flat JSON, add a hosted database, or use EF Core/DbContext - rejected by DD-CLASSIFY-STATE-STORE.
- Do NOT introduce generic repositories, provider-neutral units of work, or speculative storage interfaces - rejected by DD-CLASSIFY-APPLICATION-ARCHITECTURE.
- Do NOT persist raw corpus rows or full Ledger projection payloads.
- Do NOT permit implicit cascade deletion of referenced history - per ADR-CORE-0020.

### Expected Outputs

- ClassifyStateStore
- ClassifySchema migration chain
- ClassifyOperationIdempotencyStore
- ClassifyRowMapper

### Constraints

- Use Microsoft.Data.Sqlite and explicit caller-owned transactions; SQL behavior is verified with real temporary stores.

### Notes

- The schema is one cohesive blocker; later vertical slices own behavior-specific stores over these tables.
