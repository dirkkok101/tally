## Recipe

### Acceptance Checks

- Rule status identifies the active RuleSetVersion and draft, active, retired, and superseded rule versions with safe attributable lifecycle metadata.
- Validation and evaluation status returns retained fingerprints, lifecycle, aggregates, staleness, and safe next action without corpus or projection reread.
- Apply status returns authorized fingerprint, terminal item totals, exact unresolved frontier, mutation possibility, and whether replay or resume is safe.
- Feedback, abandonment, and cleanup status returns requested references, decision or event state, actor, reason, safe counts, and next action without payload search.
- Unknown subjects return stable not-found; expected failures name only retry, resume, re-evaluate, correct, abandon, or cleanup as allowed by durable state.

### Failure Criteria

- Do NOT reread private corpus rows or Ledger projection payloads to construct status.
- Do NOT heuristically search serialized payloads for an unknown subject.
- Do NOT claim no mutation when a planned or unresolved apply item leaves Ledger outcome unknown.
- Do NOT expose hidden rule contents, descriptions, amounts, normalized tokens, or unrelated records.

### Expected Outputs

- GetClassificationStatusQuery
- SafeNextActionPolicy
- Complete recovery registration

### Constraints

- Status is a bounded read-only projection over durable CLASSIFY state and public safe identifiers.

### Notes

- Every failure-state result contains exactly one safe next operation and explicit mutation-possibility metadata.
