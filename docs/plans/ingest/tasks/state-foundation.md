## Recipe

### Acceptance Checks

- IngestDatabase opens only <TallyDataRoot>/ingest/ingest.db, enables foreign_keys, WAL, synchronous=FULL, and bounded busy handling, and never opens or attaches LedgerDb.
- IngestMigrationV001 creates every table, key, uniqueness rule, and relationship in DM-INGEST-STATE-STORE with snake_case identifiers; PRAGMA user_version advances atomically from 0 to 1.
- Repeated migration is idempotent, unknown newer user_version fails with a stable compatibility result, and interruption leaves either the complete prior or complete V001 schema.
- IngestArtifactProtection creates directories as 0700 and state, WAL, SHM, lock, and atomic artifacts as 0600 where supported; inability to establish protection blocks persistence.
- Real temporary SQLite tests enforce immutable revision numbering, unique idempotency keys, foreign keys, and concurrent writer bounds; architecture guards find no LedgerDb or private LEDGER storage reference.

### Failure Criteria

- Do NOT share LedgerDb or LEDGER tables, attach databases, or use private LEDGER storage â€” rejected per DD-INGEST-STATE-STORE.
- Do NOT replace raw Microsoft.Data.Sqlite with EF Core, PostgreSQL, JSON batch files, or a generic repository â€” rejected per DD-INGEST-STATE-STORE and DD-INGEST-APPLICATION-ARCHITECTURE.
- Do NOT persist when owner-only permissions cannot be established â€” per DD-INGEST-ARTIFACT-SECURITY.
- Do NOT modify V001 after release; add a new immutable migration version.

### Expected Outputs

- IngestDatabase
- IngestMigrationV001 and migrator
- IngestArtifactProtection
- Real-SQLite foundation tests

- Use explicit SQL transactions and propagate CancellationToken through every asynchronous SQLite call.
