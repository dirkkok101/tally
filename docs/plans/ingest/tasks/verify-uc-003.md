## Recipe

### Acceptance Checks

- Main scenario commits the exact approved revision in frozen order through public ledger.transaction.record, round-trips each accepted transaction through ledger.transaction.get, and returns one complete receipt with exact counts/references.
- Manifest integrity, approval, account, adapter, manifest-schema, or Ledger compatibility failure creates no new mutation.
- A Ledger validation/identity conflict becomes a durable stable candidate outcome and stops before the next candidate.
- Receipt persistence failure stops further candidates and status exposes the last durable frontier.
- Interruption after Ledger commit but before receipt update resumes with the same request/key, obtains the original result, and creates no second transaction.
- A materially different retry request/key/manifest preserves the original effect and returns stable conflict; repeated resume converges.

### Failure Criteria

- Do NOT assert via LedgerDb or ingest.db; use public Ledger transaction get/list observations and INGEST receipt/status.
- Do NOT weaken fault injection or reset durable state between interruption and resume.
