## Recipe

### Acceptance Checks

- Backup requires an owner-only recognized target, actor, safe normalized path identity, and idempotency; it records one Attempting reservation with request fingerprint and deterministic artifact identity before external artifact work.
- A read-consistent candidate captures every snapshot, payload, Restatement link, DeletionEvent, prior terminal idempotency outcome, retained policy fingerprint, storage version, and count while excluding only the current non-terminal Backup reservation and all forbidden ephemeral/private data.
- InsightsBackupVerifier independently checks storage/version compatibility, owner and permissions, integrity, counts, links, payload/content/table/artifact fingerprints, terminal idempotency outcomes, deletion consistency, and policy metadata before atomic publication.
- After publication Backup records one terminal receipt; an exact retry after a crash reconciles the already published verified deterministic artifact without duplication, and a completed failure removes its reservation.
- VerifyInsightsBackupQuery is read-only: valid and invalid artifacts return exact evidence or failure, change neither active state nor Status, and expose no private path or report payload.

### Failure Criteria

- Do NOT use an unverified SQLite file copy, include the current Attempting reservation as a terminal outcome, publish before verification, or issue a second artifact on exact crash replay.
- Do NOT persist standalone Verify results, alter CURRENT, modify the source artifact, or expose private paths and payloads â€” per DD-INSIGHTS-VERIFIED-RECOVERY-ACTIVATION.
- Do NOT accept unsafe ownership, permissive modes, omission, corruption, incompatible versions, dangling links, count mismatch, or fingerprint mismatch.
