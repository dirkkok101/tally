## Recipe

### Acceptance Checks

- Delete requires explicit snapshot identity, trusted P1 actor, non-blank reason, and idempotency identity; unknown target returns NotFound and creates no event.
- A target referenced by any surviving child returns DependencyConflict and changes no payload, link, event, or replay outcome.
- A leaf deletion atomically removes its canonical payload and outgoing predecessor link and appends one DeletionEvent containing identities, actor, reason, trusted time, dependency result, and replay reference but no amount, explanation, or report content.
- Leaves can be deleted toward the root while each surviving predecessor remains immutable; exact retry of an already deleted target returns the prior metadata-only result and creates no second event.
- Existing backups remain unchanged and the result states an explicit restore may reinstate their captured state; time and limits never trigger automatic deletion.

### Failure Criteria

- Do NOT cascade-delete referenced predecessors, delete descendants, rewrite chains, return deleted payloads, or create an event for NotFound â€” per ADR-CORE-0020 and DD-INSIGHTS-RETENTION-RESTATEMENT-LIFECYCLE.
- Do NOT add automatic expiry, retention cleanup, background jobs, or backup mutation.
- Do NOT split payload, link, event, and replay changes across transactions or log deleted report content.

### Expected Outputs

- DeleteInsightReportCommand
- Dependency-safe deletion tests

### Constraints

- Deletion remains explicit, leaf-only, attributable, idempotent, and metadata-only.
