## Recipe

### Acceptance Checks

- The matrix injects faults before and after snapshot, payload, Restatement link, DeletionEvent, replay reservation/outcome, backup candidate, current-reservation removal, artifact publication, terminal receipt, restore copy, migration, candidate outcome, verification, expected-current check, pointer replacement, and directory fsync boundaries.
- After each Retain, Restate, or Delete fault and restart, the store contains either the complete logical outcome exactly once or the unchanged pre-operation state with no partial payload, dangling link, missing event, duplicate identity, or consumed failed key.
- After each Backup fault, exact replay either resumes or reconciles the deterministic artifact and leaves at most one published verified artifact and one terminal outcome; completed failure leaves no stale reservation.
- After each Restore fault, startup observes the complete prior or complete verified candidate generation, never a mixed store, and the prior generation remains recoverable.
- The report records only fault labels, safe fingerprints, row/count deltas, exit states, and test totals and contains no financial payload, private path, or report body.

### Failure Criteria

- Do NOT accept aggregate test totals without nonzero discovery for every mutation and recovery fault partition.
- Do NOT repair failed state in the test, skip restart inspection, ignore WAL/pointer/fsync boundaries, weaken exact replay, or suppress a mixed/partial outcome.
- Do NOT print fixture payloads, report bodies, private paths, or raw database rows in fault evidence.

### Expected Outputs

- InsightsAtomicRecoveryEvidence
- Reproducible failure-injection report

### Constraints

- Tests use real temporary SQLite, real file operations, and process-restart-equivalent store reopening; SQL and filesystem authority switches are not mocked.
