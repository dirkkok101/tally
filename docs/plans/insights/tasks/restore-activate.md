## Recipe

### Acceptance Checks

- Restore requires source, expected current generation fingerprint, actor, safe path identity, and idempotency and resolves an exact prior outcome before source work.
- The source is independently verified before a separate new generation is created; only compatible ordered migrations run against the candidate, and complete logical, fingerprint, permission, and integrity verification follows.
- The candidate records exactly the current Restore replay outcome and recovery event, is flushed and reverified, then activation acquires the exclusive module writer lock and compares expected current fingerprint before changing authority.
- Successful activation atomically replaces CURRENT and fsyncs its directory; startup observes either the complete old or complete new generation, Status cites durable Restore verification, and the prior generation remains a recovery point.
- Corruption, omission, incompatibility, permission, space, cancellation, migration, verification, expected-current drift, pointer replacement, or fsync failure before completed activation leaves the prior authoritative store unchanged and returns an exact stable failure.

### Failure Criteria

- Do NOT restore over the live database or rename individual database/WAL files — rejected per DD-INSIGHTS-VERIFIED-RECOVERY-ACTIVATION.
- Do NOT activate before full candidate verification, skip expected-current revalidation, overwrite the prior generation, persist standalone Verify, or add source business data beyond the captured backup.
- Do NOT include candidate paths or report payloads in results, diagnostics, logs, or crash output.

### Expected Outputs

- InsightsRestoreActivator
- RestoreInsightsBackupCommand
- Atomic activation fault tests

### Constraints

- Only CURRENT is the authority switch; directory fsync and exclusive writer coordination are mandatory.
