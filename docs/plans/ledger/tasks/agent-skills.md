## Recipe

### Acceptance Checks

- Embedded manifest declares provider, bundle checksum, operation IDs, executable range, and contract range and is generated/validated against OperationRegistry.
- skill list/check are read-only and classify compatible/missing/outdated/newer/invalid for codex|claude-code and project|user scopes.
- skill install fails before write for unsupported provider/incompatible contract/unsafe path/invalid manifest; compatible install publishes only documented provider scope then verifies checksum/version.
- Identical install replay returns original path/version/checksum with no second effect; crash after publication reconciles matching manifest.
- Bundled skills use only tally help/schema/public operations, include idempotency and safe-error workflows, and never teach private SQLite/file access.

### Failure Criteria

- Do NOT use repo-only skills, remotely fetch latest bundles, install incompatible content, bypass CLI via SQLite, or create a general plugin runtime â€” rejected per DD-LEDGER-SKILL-COMPATIBILITY.
- Do NOT duplicate descriptor metadata manually; manifest parity comes from OperationRegistry.
