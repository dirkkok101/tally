## Recipe

### Acceptance Checks

- match_existing revalidates statement evidence fingerprint, policy/version or owner authority, exact current fact equality, candidate uniqueness, conflicts, and match basis inside one BEGIN IMMEDIATE transaction before appending decision and confirming link.
- create_statement_only requires typed canonical facts equal to the normalized statement observation and atomically creates one statement-derived fact, unknown payment attribution, unassigned pool, decision, confirming link, and replay result only when no guard candidate exists.
- record_ambiguous preserves the complete candidate set and applies no confirming link or financial effect; record_exception preserves stable code/reason; stale, unsupported, conflicting, or changed replay preserves prior state.
- Crash injection at every base write boundary leaves the full outcome absent or committed; identical logical evidence and disposition replay returns the existing outcome.

### Failure Criteria

- Do NOT implement correct_existing_from_statement in this base task; TASK-LEDGER-RECONCILIATION-STATEMENT-CORRECTION owns the cross-feature composite.
- Do NOT trust the advisory token, guess among candidates, or create statement-only while a guard exists — per DD-LEDGER-RECONCILIATION-CONTRACT.
- Do NOT implement INGEST batch/manifests/previews/approvals/receipts, use transport IDs as authority, or overwrite facts/evidence/decisions.
