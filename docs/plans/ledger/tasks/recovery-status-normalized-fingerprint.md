## Recipe

### Acceptance Checks

- Given a verified current Ledger, when ledger.storage.status succeeds, then currentFingerprint and currentNormalizedFingerprint are separate lowercase 64-character SHA-256 values and currentNormalizedFingerprint equals DurableLedgerVerifier.Report.NormalizedFingerprint for that state.
- Given an existing storage-evolution caller, when it uses status.currentFingerprint as expectedCurrentFingerprint, then the existing prepare/activate lifecycle and stale-current behavior remain unchanged.
- Given schema discovery for ledger.storage.status, when the response schema is inspected, then StorageStatusResult includes currentNormalizedFingerprint as a required string field.

### Failure Criteria

- Do NOT repurpose currentFingerprint as the normalized value because storage evolution consumes its existing source-fingerprint semantics per DD-LEDGER-RECOVERY-FINGERPRINT-SURFACES.
- Do NOT weaken restore activation to compare the storage source fingerprint because activation must revalidate complete normalized Durable Ledger State per DD-LEDGER-RECOVERY-FINGERPRINT-SURFACES.
- Do NOT derive the normalized current value from backup or restore.prepare receipts because those mutations change durable idempotency state before activation per DD-LEDGER-RECOVERY-FINGERPRINT-SURFACES.

### Expected Outputs

- StorageStatusResult.CurrentNormalizedFingerprint and verified status population

### Constraints

- Preserve the 74-operation provider-neutral public inventory and existing CurrentFingerprint field.
