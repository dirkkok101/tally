## Recipe

### Acceptance Checks

- Two active ZAR transactions on distinct active owned accounts with explicit outflow/inflow roles and equal-opposite principal create one attributable active transfer.
- Different supplied dates do not block a valid explicit confirmation; a separately recorded fee stays unlinked.
- Same account/sign, unequal magnitude, different currency, inactive leg, or either leg already in any active transfer/refund returns its documented stable error and creates no relationship.
- Relationship get returns roles, exact principal, status, actor/time, and optional history while source transactions remain unchanged.
- Identical replay returns the original relationship; competing role assignment creates at most one active relationship.

### Failure Criteria

- Do NOT infer transfers from descriptions/dates, mutate transaction facts, link embedded fees, or use a generic polymorphic relationship create — per DD-LEDGER-IMMUTABLE-HISTORY.
- Do NOT implement before OQ-LEDGER-5 resolves consistently.
